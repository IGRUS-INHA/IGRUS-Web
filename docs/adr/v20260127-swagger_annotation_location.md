# Swagger 어노테이션 위치 - 컨트롤러 직접 작성

## 선택지
1. 인터페이스 분리: Swagger 어노테이션을 별도 인터페이스에 정의하고 컨트롤러가 구현
2. 컨트롤러 직접 작성: Swagger 어노테이션을 컨트롤러 클래스에 직접 작성

## 결정
- 컨트롤러 클래스에 Swagger 어노테이션 직접 작성

## 결정 이유

### 1. 단일 진실점(Single Source of Truth) 유지
- 인터페이스와 구현체가 분리되면 두 곳에 정보가 존재하게 됨
- 실제 API 동작과 Swagger 문서의 불일치 가능성 발생
- 컨트롤러에 직접 작성하면 코드와 문서가 항상 동기화됨

### 2. 정보 불일치 방지
- 인터페이스의 어노테이션과 실제 구현체의 동작이 다를 수 있음
- 파라미터 변경, 응답 타입 수정 시 인터페이스 업데이트를 누락하기 쉬움
- 단일 위치에서 관리하면 이러한 실수를 원천 차단

### 3. 개발 편의성
- 어노테이션 확인을 위해 인터페이스와 구현체를 왔다갔다하는 불편함 해소
- 컨트롤러만 보면 API의 모든 정보(로직, 문서)를 파악 가능
- 코드 리뷰 시에도 한 파일에서 전체 맥락 확인 가능

### 4. 가독성 문제는 IDE 지원으로 해결
- 컨트롤러에 어노테이션이 많아지면 가독성이 떨어지는 단점 존재
- 이 문제는 IDE의 코드 폴딩(Code Folding) 기능을 활용하여 해결
- IntelliJ IDEA 등 현대 IDE는 어노테이션 블록을 접어서 핵심 로직에 집중 가능
- 도구의 지원을 받아 단점을 보완하고 단일 진실점의 이점을 취함

## 고려한 대안

### 인터페이스 분리 방식
**장점:**
- 컨트롤러 코드가 간결해짐
- 문서화 관심사와 비즈니스 로직 분리
- 인터페이스만 보고 API 스펙 파악 가능

**채택하지 않은 이유:**
- 인터페이스와 구현체 간 정보 불일치 위험
- 변경 시 두 곳을 수정해야 하는 번거로움
- 파일 간 이동으로 인한 개발 생산성 저하
- 단일 진실점 원칙 위배

## 결과

- 모든 컨트롤러에 Swagger 어노테이션(`@Operation`, `@ApiResponse` 등) 직접 작성
- API 문서용 인터페이스는 생성하지 않음
- 코드와 문서의 일관성 보장
