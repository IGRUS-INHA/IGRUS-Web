# 프로젝트 구조 - 모노레포 사용

## 선택지
1. 모노레포 (Monorepo): 프론트엔드와 백엔드를 하나의 저장소에서 관리
2. 멀티레포 (Polyrepo): 프론트엔드와 백엔드를 별도의 저장소로 분리

## 결정
- 모노레포 구조 채택

## 결정 이유

### 1. 단순한 프로젝트 관리
- 동아리 프로젝트 특성상 소규모 팀이 프론트엔드와 백엔드를 함께 개발
- 하나의 저장소에서 전체 프로젝트 히스토리 추적 가능
- PR 하나로 프론트엔드-백엔드 변경사항을 함께 리뷰 가능

### 2. 개발 환경 설정 간소화
- 새로운 팀원이 하나의 저장소만 클론하면 전체 개발 환경 구성 가능
- 별도의 저장소 동기화 작업 불필요

### 3. API 변경 시 일관성 유지
- 백엔드 API 변경과 프론트엔드 수정을 하나의 커밋/PR로 관리
- API 계약 불일치로 인한 오류 감소

### 4. 공유 리소스 관리 용이
- API 스펙, 문서, 설정 파일 등을 한 곳에서 관리
- 프로젝트 전체에 대한 이슈 트래킹 통합

## 고려한 대안

### 멀티레포 (Polyrepo)
**장점:**
- 각 프로젝트의 독립적인 버전 관리
- 저장소별 세분화된 접근 권한 설정

**채택하지 않은 이유:**
- 레포를 분리하게 되면 각 파트 개발자 간의 작업 현황 공유가 어려움. 모든 히스토리와 작업 현황을 한 레포에서 관리하면 다른 파트에서 무엇을 하고 있는지 쉽게 확인 가능함. 협업하기 유리함.

## 결과

- 프론트엔드 코드는 `frontend/` 디렉토리에 배치
- 백엔드 코드는 `backend/` 디렉토리에 배치
- 프로젝트 문서는 `docs/` 디렉토리에 배치
